# 二进制记录方案可行性分析

> 基于 DuckDB v1.5.0 源码验证，源码目录: `source_code/`

## 结论

**可行。** 经 DuckDB v1.5.0 源码验证，扩展机制和 MultiFileReader 框架提供完整的自定义文件格式支持。已验证具体接口、函数签名和参考实现。

### 1.7 不可行的路径

DuckDB **没有任何内置功能**可以直接读写设计文档中的自定义二进制格式。原因：

1. 设计文档定义了自定义文件头（版本号+信号元信息+公式字符串），DuckDB 不认识此结构
2. 数据区位图标记稀疏数据模型，SQL 层无对应概念
3. 原始小端序二进制存储（非文本编码），`read_csv` 等于零
4. 固定长度类型的直接内存偏移读取，需要自定义 `FileHandle::Read` 调用

**只能用 C++ 扩展实现。**

### 3.2 BindReader —— 解析文件头 (基于 multi_file_function.hpp:35 接口验证)

```cpp
void BinRecordMultiFileInfo::BindReader(
    ClientContext &context,
    vector<LogicalType> &return_types,
    vector<string> &names,
    MultiFileBindData &bind_data)
{
    auto &fs = FileSystem::GetFileSystem(context);
    auto &file_path = bind_data.file_list->GetFirstFile().path;
    auto handle = fs.OpenFile(file_path, FileFlags::FILE_FLAGS_READ);

    uint64_t version;        handle->Read(&version, sizeof(version));
    uint64_t signal_count;   handle->Read(&signal_count, sizeof(signal_count));

    for (uint64_t i = 0; i < signal_count; i++) {
        uint8_t type_enum;   handle->Read(&type_enum, 1);
        uint32_t dim_count;  handle->Read(&dim_count, sizeof(dim_count));
        vector<uint32_t> dims(dim_count);
        if (dim_count > 0)
            handle->Read(dims.data(), dim_count * sizeof(uint32_t));

        uint64_t name_len;   handle->Read(&name_len, sizeof(name_len));
        string name(name_len, '\0');
        handle->Read(name.data(), name_len);

        uint32_t formula_len; handle->Read(&formula_len, sizeof(formula_len));
        if (formula_len > 0)
            handle->Seek(handle->SeekPosition() + formula_len);

        LogicalType t = MapSignalType(type_enum);
        for (int d = dim_count - 1; d >= 0; d--)
            t = LogicalType::LIST(t);

        names.push_back(name);
        return_types.push_back(t);
    }
}
```

### 3.3 BinRecordReader —— 核心扫描 (基于 base_file_reader.hpp:82 接口验证)

```cpp
class BinRecordReader : public BaseFileReader {
public:
    bool TryInitializeScan(ClientContext &, GlobalTableFunctionState &,
                           LocalTableFunctionState &) override;
    AsyncResult Scan(ClientContext &, GlobalTableFunctionState &,
                     LocalTableFunctionState &, DataChunk &) override;
    string GetReaderType() const override { return "BinRecord"; }
};

AsyncResult BinRecordReader::Scan(
    ClientContext &, GlobalTableFunctionState &gstate,
    LocalTableFunctionState &, DataChunk &output)
{
    auto &handle = file_handle;
    auto signal_count = bind_data.signal_count;
    idx_t row = 0;

    while (row < STANDARD_VECTOR_SIZE) {
        if (handle->SeekPosition() >= handle->GetFileSize()) break;

        auto bitmap_bytes = (signal_count + 7) / 8;
        vector<uint8_t> bitmap(bitmap_bytes);
        handle->Read(bitmap.data(), bitmap_bytes);

        // 时间信号 (bit 0 = 1)
        uint64_t ts; handle->Read(&ts, sizeof(ts));
        FlatVector::GetData<uint64_t>(output.data[0])[row] = ts;

        for (uint64_t s = 1; s < signal_count; s++) {
            if (bitmap[s / 8] & (1 << (s % 8)))
                ReadSignalData(handle, output.data[s], row);
            else
                FlatVector::SetNull(output.data[s], row, true);
        }
        row++;
    }
    output.SetCardinality(row);
    // AsyncResult 可从 SourceResultType 隐式构造（async_result.hpp:39）
    return row > 0 ? SourceResultType::HAVE_MORE_OUTPUT
                    : SourceResultType::FINISHED;
}
```

## 5. 难点与对策

**难点 1: 变长类型数据帧边界**

kString/kBinary 长度不固定，无法直接 seek 跨越帧。

对策：顺序读取天然支持 — 每帧先读位图、固定长度信号直接 `Read`、变长信号先读 8 字节长度前缀再读数据。JSON reader (`json_reader.cpp`) 同样逐字节流解析，模式一致。

**难点 2: 并行读取**

无帧索引无法 seek 到第 N 帧起始位置。

对策：
- 方案 A（简单）：单线程顺序读取。覆盖 `MaxThreads()` 返回 1，WRP 仿真数据量通常是 MB~GB 级，完全可行。
- 方案 B（进阶）：首次读取建立帧偏移索引，后续可利用索引并行。类似 JSON reader 的 buffer index 机制。

**难点 3: 多维数组 LIST 嵌套**

对策：`LogicalType::LIST(LogicalType::LIST(DOUBLE))` 原生支持。DuckDB JSON transform 已验证嵌套 LIST。

## 6. 设计文档待完善项

对照设计文档（`二进制记录方案设计.md`），以下要素未明确，实现时需补充约定：

| 项目 | 现状 | 建议 |
|------|------|------|
| 多维数组帧大小计算 | §4.3 仅说"按顺序连续存储"，未给数据区中一帧内数组信号的大小计算公式 | 补充：帧内数组信号数据大小 = 各维度长度乘积 × 元素字节数 |
| LIST 中变长元素 | 未说明数组元素为 kString/kBinary 时的存储格式 | 补充：每个元素按变长格式存储（8字节长度前缀+数据） |
| 位图位序 | §3.3 未说明 LSB first 还是 MSB first | 约定 LSB first（即 bit 0 对应信号0），与 `bitmap[s/8] & (1 << (s%8))` 一致 |
| 文件结束检测 | 无帧结束标记，仅靠 `SeekPosition >= FileSize` | 建议在数据区末尾添加 8 字节魔数校验或帧计数 |
| `CreateReader` 重载数量 | 设计文档未涉及，但源码中有三个重载 | 实现时需覆盖三个重载（参见 `json_multi_file_info.hpp:43-47`） |
