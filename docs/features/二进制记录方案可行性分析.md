# 二进制记录方案可行性分析

> 修订依据：DuckDB v1.5.0 源码目录 `source_code/duckdb`，以及本地 `source_code/libs/duckdb_jdbc-1.5.2.1.jar` API 检查。

## 结论

**二进制记录方案可行，但原分析需要修正。**

1. DuckDB 原生不认识本方案定义的 `.bin` 文件格式，不能直接 `SELECT * FROM 'data.bin'`。
2. C++ 扩展路线可行，但不是唯一实现路径。
3. 在最终目标为 Spring Boot 3 + `duckdb_jdbc:1.5.2.1` 的前提下，推荐主路径是：**Java 侧解析二进制文件头和数据帧，然后通过 DuckDB JDBC Appender 批量写入 DuckDB 表，再用 SQL 查询表数据**。
4. 如果纯 Java 解析性能或能力不足，可以采用混合方案：**Java/Spring Boot + C++ 动态库**。混合方案应优先让 C++ 只承担二进制解析/批处理，DuckDB 写入和查询仍通过 `duckdb_jdbc` 完成；只有需要 DuckDB 引擎内直接扫描 `.bin` 文件时，才考虑实现 DuckDB C++ 扩展。

推荐落地优先级：

| 优先级 | 方案 | 适用场景 | 结论 |
|---|---|---|---|
| 1 | Java 解析 + `DuckDBAppender` 入库 | Spring Boot 服务内导入、查询、分析 | 首选，部署简单，和现有 Java 技术栈一致 |
| 2 | Java table function `read_bin_record(path)` | 想避免预导入，直接在 SQL 中扫描文件 | 可行，但 Java 回调扫描的性能和状态管理需验证 |
| 3 | C++ `.so` 解析库 + JNI/JNA + `DuckDBAppender` | Java 解析成为瓶颈，或需要复用已有 C++ 解析器 | 可行，复杂度适中 |
| 4 | DuckDB C++ 扩展 | 需要 DuckDB 原生 `read_bin_record()`、多文件 reader、replacement scan | 可行，复杂度最高，需处理 DuckDB ABI/版本/扩展加载 |

## 1. 原文需要更正的点

### 1.1 `.bin` 文件不会被 DuckDB 自动识别

原文说“扩展注册后，DuckDB 自动支持 `SELECT * FROM 'file.ext'` 直接查询”，这个表述过强。

DuckDB 的直接文件查询依赖 replacement scan。源码中内置的文件识别是按已知后缀和已注册 replacement scan 处理的，不会对任意扩展名自动生效：

- `source_code/duckdb/src/function/table/read_csv.cpp` 只为 `.csv`、`.tsv` 注册 `ReadCSVReplacement`。
- `source_code/duckdb/src/main/database_path_and_type.cpp` 中硬编码的可直接打开后缀包含 `parquet`、`csv`、`tsv`、`json`、`jsonl`、`ndjson`，不包含 `bin`。
- `source_code/duckdb/src/main/capi/replacement_scan-c.cpp` 提供 C API 注册 replacement scan 的机制，但 `duckdb_jdbc` 没有直接暴露 `duckdb_add_replacement_scan` 这种底层入口。

因此：

```sql
SELECT * FROM 'data.bin';
```

在当前方案下不成立。需要以下之一：

- 显式调用 Java 注册的 table function，例如 `SELECT * FROM read_bin_record('data.bin')`。
- 先由 Java 导入成 DuckDB 表，再 `SELECT * FROM sim_data`。
- 实现并加载 DuckDB C++ 扩展，在扩展中注册 table function/replacement scan。

### 1.2 “只能用 C++ 扩展实现”不正确

DuckDB 内核确实没有内置读取本二进制格式的能力。若目标是让 DuckDB 引擎直接扫描 `.bin` 文件，C++ 扩展是合理路线。

但最终实现目标是 Spring Boot 3 项目，并已指定 JDBC 依赖：

```xml
<dependency>
    <groupId>org.duckdb</groupId>
    <artifactId>duckdb_jdbc</artifactId>
    <version>1.5.2.1</version>
    <scope>compile</scope>
</dependency>
```

在这个目标下，不需要先做 DuckDB C++ 扩展。`duckdb_jdbc:1.5.2.1` 已具备以下能力：

- `DuckDBConnection.createAppender(...)`
- `DuckDBAppender`
- `DuckDBFunctions.tableFunction()`
- `DuckDBTableFunctionBuilder`
- `DuckDBTableFunction`
- `DuckDBDataChunkWriter`

这些类在本地 `source_code/libs/duckdb_jdbc-1.5.2.1.jar` 中存在，并通过 `javap` 验证。`DuckDBAppender` 支持基本类型、字符串、BLOB、数组/集合、Map、Struct、NULL 等写入能力，足以覆盖本方案的主要类型映射。

### 1.3 `MultiFileReaderInterface` 的必须实现项表述需修正

源码 `source_code/duckdb/src/include/duckdb/common/multi_file/multi_file_function.hpp` 中，真正纯虚、必须实现的是：

| 方法 | 结论 |
|---|---|
| `InitializeOptions()` | 必须 |
| `ParseCopyOption()` | 必须 |
| `ParseOption()` | 必须 |
| `InitializeBindData()` | 必须 |
| `BindReader()` | 必须 |
| `InitializeGlobalState()` | 必须 |
| `InitializeLocalState()` | 必须 |
| `CreateReader(ClientContext&, GlobalTableFunctionState&, BaseUnionData&, ...)` | 必须 |
| `CreateReader(ClientContext&, GlobalTableFunctionState&, const OpenFileInfo&, idx_t, ...)` | 必须 |

以下不是纯虚函数：

| 方法 | 修正 |
|---|---|
| `CreateReader(ClientContext&, const OpenFileInfo&, BaseFileReaderOptions&, const MultiFileOptions&)` | 有默认实现，不是必须覆盖；如果走到这条路径且未覆盖会抛 `InternalException` |
| `MaxThreads()` | 有默认实现，返回空 `optional_idx()`；顺序扫描时可以覆盖返回 `1`，但不是强制 |
| `GetGlobInput()` | 有默认实现；只有需要允许空 glob 或指定默认后缀时才覆盖 |
| `GetCardinality()` | 可选估算 |
| `FinishReading()`、`Copy()`、`GetVirtualColumns()` | 视能力选择 |

原文“`CreateReader()` 三个重载都需全部覆盖”不准确。

### 1.4 `BaseFileReader` 接口描述基本正确

源码 `source_code/duckdb/src/include/duckdb/common/multi_file/base_file_reader.hpp` 中，必须实现的纯虚函数是：

- `TryInitializeScan(...)`
- `Scan(...)`
- `GetReaderType()`

以下是可选覆盖：

- `PrepareReader(...)`
- `PrepareScan(...)`
- `FinishFile(...)`
- `GetProgressInFile(...)`
- `GetStatistics(...)`
- `AddVirtualColumn(...)`

### 1.5 bind 阶段不应被描述成“固定打开文件并扫描”

DuckDB table function 的 bind 阶段负责解析参数、确定输出 schema、生成 bind data。对于本格式，如果 DuckDB 扩展路线需要从文件头推导列名和类型，bind 阶段可以打开首个文件读取 header。

但这不是 DuckDB 对 table function 的普遍要求，也不是 Java 落地时的唯一方式。Java/Spring Boot 主路径中，header 解析应属于应用层 parser/importer；DuckDB 只接收建表 SQL 和 Appender 写入。

### 1.6 “过滤下推自动获得”需要降级为条件能力

`MultiFileFunction` 会设置 `projection_pushdown = true`，列裁剪能力相对容易获得。但复杂过滤下推不是“自动完成业务语义”的能力。对二进制帧格式而言：

- 无帧索引时，`time > x` 仍然需要顺序读到对应帧，除非额外建立时间索引。
- 非时间列过滤通常无法跳过读取整帧，只能在 DuckDB 层过滤。
- Java Appender 入库路线中，下推问题不存在，因为查询对象已经是 DuckDB 表。

## 2. DuckDB C++ 路线的源码证据

如果后续决定做 DuckDB C++ 扩展，原文识别的总体方向是正确的。

### 2.1 表函数接口

源码：`source_code/duckdb/src/include/duckdb/function/table_function.hpp`

关键能力：

- bind：确定输出列名和类型。
- init global/local：初始化全局和线程本地扫描状态。
- scan：向 `DataChunk` 输出批量数据。
- `projection_pushdown`：列裁剪开关。
- `filter_pushdown` / `pushdown_complex_filter`：条件下推相关开关和回调。

### 2.2 MultiFileReader 框架

源码：

- `source_code/duckdb/src/include/duckdb/common/multi_file/base_file_reader.hpp`
- `source_code/duckdb/src/include/duckdb/common/multi_file/multi_file_function.hpp`
- `source_code/duckdb/src/common/multi_file/multi_file_function.cpp`
- `source_code/duckdb/src/include/duckdb/common/multi_file/multi_file_reader.hpp`

该框架适合实现 `read_bin_record('data_*.bin')` 这种多文件扫描函数。可复用的能力包括：

- 文件列表解析和 glob 展开。
- bind 阶段统一处理。
- 全局/本地状态初始化。
- 多文件调度。
- 进度和基数估算接口。
- filename、hive partition 等虚拟列能力。

### 2.3 参考实现

可参考：

- `source_code/duckdb/extension/json/include/json_multi_file_info.hpp`
- `source_code/duckdb/extension/json/json_multi_file_info.cpp`

- `source_code/duckdb/src/function/table/read_csv.cpp`

需要注意：JSON/CSV 的自动识别机制来自它们显式注册的 replacement scan，不是 MultiFileReader 自动赋予任意后缀的能力。

## 3. Java/Spring Boot 主路径可行性

### 3.1 建议架构

```text
Spring Boot API / Job
        |
        v
BinaryRecordHeaderReader      读取 magic/version/schema/signals
        |
        v
BinaryRecordFrameReader       顺序读取 frame，处理 bitmap 和 NULL
        |
        v
DuckDbSchemaMapper            映射成 DuckDB SQL 类型和 CREATE TABLE
        |
        v
DuckDbImportService           DuckDBConnection + DuckDBAppender 批量写入
        |
        v
DuckDB table                  标准 SQL 查询、聚合、导出
```

该路线不依赖 DuckDB 扩展机制，也不要求 DuckDB 认识 `.bin` 文件。

### 3.2 JDBC 能力边界

`duckdb_jdbc:1.5.2.1` 中已验证：

| 能力 | API | 结论 |
|---|---|---|
| 创建 Appender | `DuckDBConnection.createAppender(...)` | 支持 |
| 行式批量写入 | `DuckDBAppender.beginRow/endRow/flush` | 支持 |
| NULL 写入 | `DuckDBAppender.appendNull()` | 支持 |
| 基本类型写入 | `append(int/long/float/double/String/...)` | 支持 |
| BLOB | `appendByteArray(...)` / `append(byte[])` | 支持 |
| 数组/集合 | `append(int[])`、`append(double[][])`、`append(Collection<?>)` 等 | 支持 |
| Struct | `beginStruct()` / `endStruct()` | 支持 |
| Java table function | `DuckDBFunctions.tableFunction()` + `DuckDBTableFunctionBuilder` | 支持 |
| C++ 扩展加载 | 通过 JDBC `Statement` 执行 `LOAD`/`INSTALL` SQL 可尝试；未看到 `DuckDBConnection.loadExtension()` 直接 Java 方法 | 需单独验证部署和签名策略 |

### 3.3 推荐数据写入方式

1. 解析文件头，得到信号列表。
2. 生成 DuckDB 表结构。
3. 使用 `DuckDBConnection.createAppender(tableName)`。
4. 每读一帧调用：
   - `beginRow()`
   - 对 bitmap 为 1 的信号 append 对应值。
   - 对 bitmap 为 0 的信号 appendNull。
   - `endRow()`
5. 每批 flush，任务结束 close。

### 3.4 Java table function 作为可选增强

如果希望直接在 SQL 中使用：

```sql
SELECT * FROM read_bin_record('/data/sim.bin');
```

可用 `DuckDBFunctions.tableFunction()` 注册 Java table function。其适用边界：

- bind 阶段解析 header 并添加输出列。
- init 阶段打开文件并初始化 reader。
- apply 阶段写入 `DuckDBDataChunkWriter`。

风险：

- Java table function 的高吞吐性能需要压测。
- 文件句柄、异常传播、并发扫描状态要仔细实现。
- 它不等价于 replacement scan，不能自动让 `SELECT * FROM 'data.bin'` 成立。

## 4. C++ 动态库/混合方案

用户已明确 Spring Boot 项目允许使用 C++ 编译的动态库（`.so`）。因此混合方案可作为性能或能力升级路径。

### 4.1 推荐的混合方式：C++ 解析库 + Java Appender

```text
Spring Boot
    |
    | JNI/JNA/FFM
    v
libbinrecord.so          只负责 header/frame 解析和批量解码
    |
    v
Java batch objects / direct buffer
    |
    v
DuckDBAppender           仍由 duckdb_jdbc 写入 DuckDB
```

优点：

- 不需要实现 DuckDB 扩展 ABI。
- Spring Boot 仍只管理一个 DuckDB JDBC 连接。
- C++ 可以优化二进制解析、SIMD、零拷贝批处理。
- 失败时可回退到纯 Java parser。

缺点：

- 需要 JNI/JNA/FFM 封装。
- 需要发布不同平台的 `.so`。
- 需要额外处理 native 崩溃和内存生命周期。

### 4.2 DuckDB C++ 扩展方式

如果目标升级为 DuckDB 引擎内原生扫描：

```sql
LOAD '/path/bin_record.duckdb_extension';
SELECT * FROM read_bin_record('/data/sim.bin');
```

这时才需要实现原文中的 `BaseFileReader`、`MultiFileReaderInterface` 和扩展入口。该路线要额外处理：

- 扩展与 DuckDB/JDBC 内嵌 DuckDB 版本兼容。
- 扩展签名或 `allow_unsigned_extensions` 配置。
- Linux `.so`/`.duckdb_extension` 发布。
- Spring Boot 容器内 native library 路径。
- replacement scan 是否真的需要；通常 `read_bin_record(path)` 已足够。

## 5. 二进制格式到 DuckDB 的类型映射

以下是实现约定，不是 DuckDB 对文件格式的原生保证。

| 二进制类型 | 推荐 DuckDB 类型 | Java/JDBC 写入建议 |
|---|---|---|
| `kUint64` | `UBIGINT` | `BigInteger` 或确认无符号范围后 `long` |
| `kUint32` | `UINTEGER` | `long`/`int` 按范围处理 |
| `kUint16` | `USMALLINT` | `int`/`short` 按范围处理 |
| `kUint8` | `UTINYINT` | `short`/`byte` 按范围处理 |
| `kInt64` | `BIGINT` | `long` |
| `kInt32` | `INTEGER` | `int` |
| `kInt16` | `SMALLINT` | `short` |
| `kInt8` | `TINYINT` | `byte` |
| `kBool` | `BOOLEAN` | `boolean` |
| `kEnum` | `BIGINT` | 先按整数存储；如需名称字典，另加元数据表 |
| `kFloat` | `FLOAT` | `float` |
| `kDouble` | `DOUBLE` | `double` |
| `kString` | `VARCHAR` | `String`，UTF-8 解码 |
| `kBinary` | `BLOB` | `byte[]` |
| `kComplex32` | `STRUCT(re FLOAT, im FLOAT)` 或 `FLOAT[2]` | 优先 `STRUCT`，如 Appender 复杂度高则降级为数组 |
| `kComplex64` | `STRUCT(re DOUBLE, im DOUBLE)` 或 `DOUBLE[2]` | 同上 |
| 数组 | `LIST`/嵌套 `LIST`，或固定维度 `ARRAY` | 首版建议 `LIST`，维度存元数据表 |

补充说明：

- `kEnum` 映射为 `BIGINT` 是实现约定，不是 DuckDB ENUM。若要 DuckDB ENUM，需要在文件头或外部元数据中保存枚举字典。
- 多维数组如果用 `LIST`，DuckDB 不会自动知道固定 shape；需要保留维度元数据。
- 固定维度数组可考虑 DuckDB `ARRAY`，但首版应以 JDBC Appender 验证成本最低的 `LIST` 为准。
- 复数类型可用 `STRUCT(re, im)` 保留语义；若 Java Appender 写 struct 复杂，可以先写成长度为 2 的数组并在元数据中标记。

## 6. 设计文档必须补充的格式细节

`docs/features/二进制记录方案设计.md` 需要补齐以下内容，否则实现时会出现歧义：

| 项目 | 原文现状 | 必须补充 |
|---|---|---|
| 文件魔数 | 未定义 | 建议增加 8 字节 magic，防止误读普通 `.bin` |
| 位图位序 | 未定义 | 明确 LSB first：`bit i = bitmap[i / 8] & (1 << (i % 8))` |
| EOF/坏文件识别 | 只靠文件大小 | 增加帧长度或帧计数；至少规定 EOF 必须落在帧边界 |
| 帧边界 | 变长字段导致无法快速跳过 | 建议每帧前增加 `frame_size` |
| 变长数组元素 | 未说明 | 数组中每个 `kString`/`kBinary` 元素也按 `uint64 length + bytes` 存储 |
| 数组 NULL 语义 | 未说明 | bitmap 控制整个信号是否为 NULL；首版不支持数组内部元素 NULL |
| 多字节字节序 | 只在实现要点提到 | 所有 header、frame、长度字段统一 little endian |
| 名称合法性 | 未说明 | Java 建表时需要转义或规范化列名，原始名称保留到元数据 |
| 版本兼容 | 只有版本号 | 明确 v1 解析规则和未知版本拒绝策略 |
| 写入完成状态 | 未说明 | 规定 close 时回填帧数，异常中断文件按 best-effort 解析或拒绝 |

## 7. 推荐下一步设计

正式设计文档应以 Java/Spring Boot 落地为主，不应以 DuckDB C++ 扩展为主。

建议 `docs/features/二进制记录方案设计.md` 明确：

1. 二进制文件格式 v1。
2. Java parser 模块边界。
3. DuckDB schema 映射规则。
4. `DuckDBAppender` 导入流程。
5. 可选 Java table function。
6. 可选 C++ `.so` 解析库和 DuckDB C++ 扩展升级路径。
7. 测试样例和兼容性策略。

这样可以先以较低工程成本完成可用版本，再根据压测结果决定是否引入 C++ 动态库或 DuckDB 扩展。
