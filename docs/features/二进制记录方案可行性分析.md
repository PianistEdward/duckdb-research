# 二进制记录方案可行性分析

> 修订依据：`docs/features/二进制记录协议标准.md`、DuckDB v1.5.0 源码目录 `source_code/duckdb`，以及本地 `source_code/libs/duckdb_jdbc-1.5.2.1.jar` API 检查。

## 结论

**二进制记录方案可行，但实现必须以当前协议标准为边界。**

1. DuckDB 原生不认识本方案定义的 `.bin` 文件格式，不能直接 `SELECT * FROM 'data.bin'`。
2. 在 Spring Boot 3 + `duckdb_jdbc:1.5.2.1` 前提下，首选路线是：Java 解析协议标准文件，然后通过 DuckDB JDBC Appender 批量写入 DuckDB 表。
3. C++ 扩展路线可行，但不是首版必需路径。
4. 如果纯 Java 解析性能不足，可以采用混合方案：C++ 动态库只承担二进制解析/批处理，DuckDB 写入和查询仍通过 `duckdb_jdbc` 完成。
5. 当前协议标准未定义 magic、header_size、frame_size、frame_count 等字段；这些可以作为协议优化建议，但不能写入当前实现假设。

推荐落地优先级：

| 优先级 | 方案 | 适用场景 | 结论 |
|---|---|---|---|
| 1 | Java 解析 + `DuckDBAppender` 入库 | Spring Boot 服务内导入、查询、分析 | 首选，部署简单，和现有 Java 技术栈一致 |
| 2 | Java table function `read_bin_record(path)` | 想避免预导入，直接在 SQL 中扫描文件 | 可行，但性能和状态管理需验证 |
| 3 | C++ `.so` 解析库 + JNI/JNA/FFM + `DuckDBAppender` | Java 解析成为瓶颈，或需要复用已有 C++ 解析器 | 可行，复杂度适中 |
| 4 | DuckDB C++ 扩展 | 需要 DuckDB 原生 `read_bin_record()`、多文件 reader、replacement scan | 可行，复杂度最高，需处理 DuckDB ABI/版本/扩展加载 |

## 1. 协议标准对实现的约束

当前协议标准定义：

1. 文件头只包含 `version`、`signal_count` 和 `SignalMeta[]`。
2. 每个信号元信息包含类型、维度、名称和公式。
3. 数据区由按时间顺序追加的数据帧组成。
4. 每帧以信号存在位图开始，位图使用 LSB first。
5. 时间信号固定为第 0 个信号，每帧都有数据。
6. 所有时间帧之后写入 8 字节 EOF 标志 `0xDEADBEEFDEADBEEF`。
7. 除 `kString` 和 `kBinary` 外，其他类型为固定长度。
8. 数组信号仅支持固定长度数据类型。
9. 多字节字段使用 Little Endian，字符串使用 UTF-8。

实现影响：

- 解析器不能依赖 magic 判断文件类型，只能依赖业务入口和版本字段校验。
- 解析器不能依赖 header_size 快速定位数据区，只能按元信息结构顺序读完文件头。
- 解析器不能依赖 frame_size 跳帧，只能按 bitmap、信号类型和维度逐字段消费。
- 解析器不能依赖 frame_count 预估总帧数，只能读取到帧边界 EOF 为止。
- 当前协议下，文件损坏定位能力有限，默认应拒绝导入异常文件。

## 2. DuckDB 原生读取能力边界

### 2.1 `.bin` 文件不会被 DuckDB 自动识别

DuckDB 的直接文件查询依赖内置 reader 或 replacement scan。当前 DuckDB 不会对任意 `.bin` 文件自动调用本方案解析器。

因此：

```sql
SELECT * FROM 'data.bin';
```

在当前方案下不成立。需要以下之一：

- Java 先导入成 DuckDB 表，再 `SELECT * FROM sim_data`。
- 显式调用 Java 注册的 table function，例如 `SELECT * FROM read_bin_record('data.bin')`。
- 实现并加载 DuckDB C++ 扩展，在扩展中注册 table function 或 replacement scan。

### 2.2 “只能用 C++ 扩展实现”不成立

如果目标是让 DuckDB 引擎直接扫描 `.bin` 文件，C++ 扩展是合理路线。

但最终实现目标是 Spring Boot 3 项目，并已指定 JDBC 依赖：

```xml
<dependency>
    <groupId>org.duckdb</groupId>
    <artifactId>duckdb_jdbc</artifactId>
    <version>1.5.2.1</version>
    <scope>compile</scope>
</dependency>
```

在这个目标下，不需要首版实现 DuckDB C++ 扩展。`duckdb_jdbc:1.5.2.1` 已具备以下能力：

- `DuckDBConnection.createAppender(...)`
- `DuckDBAppender`
- `DuckDBFunctions.tableFunction()`
- `DuckDBTableFunctionBuilder`
- `DuckDBTableFunction`
- `DuckDBDataChunkWriter`

`DuckDBAppender` 支持基本类型、字符串、BLOB、数组/集合、Map、Struct、NULL 等写入能力，足以覆盖当前协议标准的主要类型映射。

## 3. Java/Spring Boot 主路径可行性

### 3.1 建议架构

```text
Spring Boot API / Job
        |
        v
BinaryRecordHeaderReader      读取 version/signal_count/signals
        |
        v
BinaryRecordFrameReader       顺序读取 frame，处理 bitmap、NULL 和 EOF
        |
        v
DuckDbSchemaMapper            映射成 DuckDB SQL 类型和 CREATE TABLE
        |
        v
DuckDbImportService           DuckDBConnection + DuckDBAppender 批量写入
        |
        v
DuckDB table                  标准 SQL 查询、聚合、导出
```

该路线不依赖 DuckDB 扩展机制，也不要求 DuckDB 认识 `.bin` 文件。

### 3.2 JDBC 能力边界

`duckdb_jdbc:1.5.2.1` 中已验证：

| 能力 | API | 结论 |
|---|---|---|
| 创建 Appender | `DuckDBConnection.createAppender(...)` | 支持 |
| 行式批量写入 | `DuckDBAppender.beginRow/endRow/flush` | 支持 |
| NULL 写入 | `DuckDBAppender.appendNull()` | 支持 |
| 基本类型写入 | `append(int/long/float/double/String/...)` | 支持 |
| BLOB | `appendByteArray(...)` / `append(byte[])` | 支持 |
| 数组/集合 | `append(int[])`、`append(double[][])`、`append(Collection<?>)` 等 | 支持 |
| Struct | `beginStruct()` / `endStruct()` | 支持 |
| Java table function | `DuckDBFunctions.tableFunction()` + `DuckDBTableFunctionBuilder` | 支持 |
| C++ 扩展加载 | 通过 JDBC `Statement` 执行 `LOAD`/`INSTALL` SQL 可尝试；未看到 `DuckDBConnection.loadExtension()` 直接 Java 方法 | 需单独验证部署和签名策略 |

### 3.3 推荐数据写入方式

1. 解析文件头，得到信号列表。
2. 生成 DuckDB 表结构。
3. 使用 `DuckDBConnection.createAppender(tableName)`。
4. 每读一帧调用：
   - `beginRow()`
   - 对 bitmap 为 1 的信号 append 对应值。
   - 对 bitmap 为 0 的信号 appendNull。
   - `endRow()`
5. 每批 flush，任务结束 close。

### 3.4 当前协议下的读取风险

当前协议标准可实现，但对健壮性有以下影响：

| 风险 | 原因 | 首版处理 |
|---|---|---|
| 不能快速验证文件类型 | 无 magic | 由业务入口限定文件来源，校验 version 和结构 |
| 不能 O(1) 定位数据区 | 无 header_size | 按信号元信息顺序读取到数据区 |
| 不能快速跳过损坏帧 | 无 frame_size | 默认拒绝损坏文件，不做跳帧恢复 |
| 不能预估总帧数 | 无 frame_count | 导入进度按已读字节估算 |
| EOF 判定依赖帧边界 | EOF 是固定 8 字节标志 | 仅在下一帧开始位置判断 EOF，且 EOF 后应到达文件末尾 |
| 变长字段可导致越界 | `kString`/`kBinary` 有长度前缀 | 校验 `data_length` 不超过剩余文件大小 |

这些风险不阻塞首版落地，但应在导入服务中显式处理并测试。

### 3.5 Java table function 作为可选增强

如果希望直接在 SQL 中使用：

```sql
SELECT * FROM read_bin_record('/data/sim.bin');
```

可用 `DuckDBFunctions.tableFunction()` 注册 Java table function。其适用边界：

- bind 阶段解析 header 并添加输出列。
- init 阶段打开文件并初始化 reader。
- apply 阶段写入 `DuckDBDataChunkWriter`。

风险：

- Java table function 的高吞吐性能需要压测。
- 文件句柄、异常传播、并发扫描状态要仔细实现。
- 它不等价于 replacement scan，不能自动让 `SELECT * FROM 'data.bin'` 成立。

## 4. C++ 动态库/混合方案

Spring Boot 项目允许使用 C++ 编译的动态库（`.so`）时，混合方案可作为性能或能力升级路径。

### 4.1 推荐的混合方式：C++ 解析库 + Java Appender

```text
Spring Boot
    |
    | JNI/JNA/FFM
    v
libbinrecord.so          只负责 header/frame/EOF 解析和批量解码
    |
    v
Java batch objects / direct buffer
    |
    v
DuckDBAppender           仍由 duckdb_jdbc 写入 DuckDB
```

优点：

- 不需要实现 DuckDB 扩展 ABI。
- Spring Boot 仍只管理一个 DuckDB JDBC 连接。
- C++ 可以优化二进制解析、SIMD、零拷贝批处理。
- 失败时可回退到纯 Java parser。

缺点：

- 需要 JNI/JNA/FFM 封装。
- 需要发布不同平台的 `.so`。
- 需要额外处理 native 崩溃和内存生命周期。

### 4.2 DuckDB C++ 扩展方式

如果目标升级为 DuckDB 引擎内原生扫描：

```sql
LOAD '/path/bin_record.duckdb_extension';
SELECT * FROM read_bin_record('/data/sim.bin');
```

这时才需要实现 `BaseFileReader`、`MultiFileReaderInterface` 和扩展入口。该路线要额外处理：

- 扩展与 DuckDB/JDBC 内嵌 DuckDB 版本兼容。
- 扩展签名或 `allow_unsigned_extensions` 配置。
- Linux `.so`/`.duckdb_extension` 发布。
- Spring Boot 容器内 native library 路径。
- replacement scan 是否真的需要；通常 `read_bin_record(path)` 已足够。

## 5. 二进制格式到 DuckDB 的类型映射

以下是实现约定，不是 DuckDB 对文件格式的原生保证。

| 二进制类型 | 推荐 DuckDB 类型 | Java/JDBC 写入建议 |
|---|---|---|
| `kUint64` | `UBIGINT` | `BigInteger` 或确认无符号范围后 `long` |
| `kUint32` | `UINTEGER` | `long`/`int` 按范围处理 |
| `kUint16` | `USMALLINT` | `int`/`short` 按范围处理 |
| `kUint8` | `UTINYINT` | `short`/`byte` 按范围处理 |
| `kInt64` | `BIGINT` | `long` |
| `kInt32` | `INTEGER` | `int` |
| `kInt16` | `SMALLINT` | `short` |
| `kInt8` | `TINYINT` | `byte` |
| `kBool` | `BOOLEAN` | `boolean` |
| `kEnum` | `BIGINT` | 先按整数存储；如需名称字典，另加元数据表 |
| `kFloat` | `FLOAT` | `float` |
| `kDouble` | `DOUBLE` | `double` |
| `kString` | `VARCHAR` | `String`，UTF-8 解码 |
| `kBinary` | `BLOB` | `byte[]` |
| `kComplex32` | `STRUCT(re FLOAT, im FLOAT)` 或 `FLOAT[2]` | 优先 `STRUCT`，如 Appender 复杂度高则降级为数组 |
| `kComplex64` | `STRUCT(re DOUBLE, im DOUBLE)` 或 `DOUBLE[2]` | 同上 |
| 固定长度数组 | `LIST`/嵌套 `LIST`，或固定维度 `ARRAY` | 首版建议 `LIST`，维度存元数据表 |

补充说明：

- `kEnum` 映射为 `BIGINT` 是实现约定，不是 DuckDB ENUM。若要 DuckDB ENUM，需要在文件头或外部元数据中保存枚举字典。
- 多维数组如果用 `LIST`，DuckDB 不会自动知道固定 shape；需要保留维度元数据。
- 固定维度数组可考虑 DuckDB `ARRAY`，但首版应以 JDBC Appender 验证成本最低的 `LIST` 为准。
- 复数类型可用 `STRUCT(re, im)` 保留语义；若 Java Appender 写 struct 复杂，可以先写成长度为 2 的数组并在元数据中标记。

## 6. DuckDB C++ 路线的源码证据

如果后续决定做 DuckDB C++ 扩展，原分析识别的总体方向仍然成立，但它属于高复杂度升级路线。

### 6.1 表函数接口

源码：`source_code/duckdb/src/include/duckdb/function/table_function.hpp`

关键能力：

- bind：确定输出列名和类型。
- init global/local：初始化全局和线程本地扫描状态。
- scan：向 `DataChunk` 输出批量数据。
- `projection_pushdown`：列裁剪开关。
- `filter_pushdown` / `pushdown_complex_filter`：条件下推相关开关和回调。

### 6.2 MultiFileReader 框架

源码：

- `source_code/duckdb/src/include/duckdb/common/multi_file/base_file_reader.hpp`
- `source_code/duckdb/src/include/duckdb/common/multi_file/multi_file_function.hpp`
- `source_code/duckdb/src/common/multi_file/multi_file_function.cpp`
- `source_code/duckdb/src/include/duckdb/common/multi_file/multi_file_reader.hpp`

该框架适合实现 `read_bin_record('data_*.bin')` 这种多文件扫描函数。可复用能力包括：

- 文件列表解析和 glob 展开。
- bind 阶段统一处理。
- 全局/本地状态初始化。
- 多文件调度。
- 进度和基数估算接口。
- filename、hive partition 等虚拟列能力。

### 6.3 必须实现项修正

`MultiFileReaderInterface` 中真正纯虚、必须实现的是：

| 方法 | 结论 |
|---|---|
| `InitializeOptions()` | 必须 |
| `ParseCopyOption()` | 必须 |
| `ParseOption()` | 必须 |
| `InitializeBindData()` | 必须 |
| `BindReader()` | 必须 |
| `InitializeGlobalState()` | 必须 |
| `InitializeLocalState()` | 必须 |
| `CreateReader(ClientContext&, GlobalTableFunctionState&, BaseUnionData&, ...)` | 必须 |
| `CreateReader(ClientContext&, GlobalTableFunctionState&, const OpenFileInfo&, idx_t, ...)` | 必须 |

`BaseFileReader` 必须实现的纯虚函数是：

- `TryInitializeScan(...)`
- `Scan(...)`
- `GetReaderType()`

## 7. 过滤下推与性能判断

对 Java Appender 入库路线，查询对象已经是 DuckDB 表，过滤和聚合由 DuckDB 正常优化，不存在外部文件 reader 的下推问题。

对 Java table function 或 C++ 扩展直接扫描路线：

1. 列裁剪相对容易，因为 bitmap 和信号定义允许只 materialize 需要的列。
2. 无帧索引时，`time > x` 仍然需要顺序读到对应帧，除非额外建立时间索引。
3. 非时间列过滤通常无法跳过读取整帧，只能在 DuckDB 层过滤。
4. 当前协议没有 `frame_size`，即使某列不需要，也仍需按协议消费不需要字段才能到达下一帧；变长字段尤其如此。

因此，首版性能目标应先以 Appender 批量导入吞吐和导入后查询性能为主，不应承诺外部文件扫描的复杂过滤下推。

## 8. 当前标准下的测试重点

必须覆盖：

1. 只含时间信号的最小文件。
2. 所有基本类型各一列。
3. 稀疏帧：bitmap 为 0 的信号导入为 `NULL`。
4. `kString`、`kBinary` 的空值和非空值。
5. 一维数组、二维数组，且元素类型为固定长度类型。
6. 数组信号错误使用 `kString` 或 `kBinary` 时拒绝导入。
7. `kComplex32`、`kComplex64` 读写顺序。
8. 位图 LSB first 规则。
9. EOF 正常结束、EOF 缺失、EOF 不在帧边界。
10. 文件截断、长度字段越界、维度乘积溢出。
11. 非法 UTF-8 策略。
12. 重复、空白、关键字等非法 SQL 列名规范化。
13. 大文件批量导入性能。

## 9. 协议优化建议处理

magic、header_size、frame_size、frame_count、文件级 flags 等字段能显著提升格式自描述性、损坏检测、跳帧能力和导入进度估算。

但这些字段不属于当前协议标准，不能直接写入当前实现设计。相关建议已单独整理到：

`docs/features/二进制记录协议标准优化建议.md`

后续只有在协议标准升级后，设计和实现才能把这些字段作为必需字段处理。
