# 二进制记录协议标准优化建议

> 本文档只记录对 `docs/features/二进制记录协议标准.md` 的后续优化建议，不改变当前协议标准。当前实现仍以协议标准原文为准。

## 1. 背景

当前协议标准已经能支持二进制记录、稀疏帧、固定长度数组信号和 EOF 结束标志。但从文件识别、损坏检测、跳帧、导入进度、EOF 歧义和版本兼容角度看，仍有可优化空间。

这些优化涉及协议字段变更，应作为下一版协议讨论，不能直接混入当前实现。

## 2. 建议增加文件魔数

建议在文件开头增加 8 字节 magic，例如：

| 字段 | 大小 | 类型 | 说明 |
|---|---:|---|---|
| magic | 8 | bytes | 固定为 `BINREC01` |

收益：

- 快速拒绝非本协议 `.bin` 文件。
- 避免仅凭扩展名或版本号误读普通二进制文件。

兼容影响：

- 会改变文件头布局。
- 需要通过新版本号区分旧格式和新格式；当前协议的 `version` 是 8 字节 `uint64`，新版本应继续保持该字段宽度，除非明确做一次破坏性升级。

## 3. 建议先做不改变协议语义的文档澄清

以下修改不改变二进制布局，可优先作为协议标准文档的勘误项：

1. 删除“设计目标”下的异常空 bullet。
2. 将“文件整体分为两部分”改为“文件头、数据帧区、EOF 标志三部分”，避免遗漏 EOF。
3. 修正数据类型章节重复的 `4.1` 小节编号。
4. 明确数据帧中 bitmap 后固定 8 字节时间值属于第 0 个信号，后续变长 `signal_data` 从信号 1 开始。
5. 明确 EOF 标志只能在帧边界、作为文件最后 8 字节出现，不能在文件任意位置扫描。

这些属于文档表达修正，不应改变现有 reader/writer 的二进制兼容性。

## 4. 建议增加 header_size

建议文件头增加 `header_size`：

| 字段 | 大小 | 类型 | 说明 |
|---|---:|---|---|
| header_size | 8 | uint64 | 从文件开头到数据区开始的总字节数 |

收益：

- 快速定位数据区。
- 校验信号元信息读取是否越界。
- 便于后续在文件头增加扩展字段。

兼容影响：

- 需要明确 `header_size` 是否包含固定文件头和所有 `SignalMeta`。

## 5. 建议增加 frame_size

建议每帧开头增加 `frame_size`：

| 字段 | 大小 | 类型 | 说明 |
|---|---:|---|---|
| frame_size | 8 | uint64 | 本帧总大小，包含 `frame_size` 字段自身 |

收益：

- 可校验帧边界。
- 可跳过不需要解析的帧。
- 损坏文件定位更清晰。
- 对变长字段更安全。

兼容影响：

- 会改变数据帧布局。
- 需要定义 `frame_size` 和 EOF 的相对位置。

## 6. 建议增加 frame_count

建议文件头增加 `frame_count`：

| 字段 | 大小 | 类型 | 说明 |
|---|---:|---|---|
| frame_count | 8 | uint64 | 已写入帧数，流式写入时可关闭文件前回填 |

收益：

- 导入前可预估行数。
- 导入进度更准确。
- 可与实际解析帧数交叉校验。

兼容影响：

- 流式写入需要支持回填。
- 异常中断文件需要定义 `frame_count` 不可信时的处理策略。

## 7. 建议增加 flags 和 reserved

建议文件级增加：

| 字段 | 大小 | 类型 | 说明 |
|---|---:|---|---|
| flags | 4 | uint32 | 文件级能力标志 |
| reserved | 32 | bytes | 预留，写 0 |

可选 flags：

- 是否压缩。
- 是否包含帧索引。
- 是否包含校验和。
- 是否允许数组内变长元素。

收益：

- 保留协议演进空间。
- 避免后续通过版本号承载所有行为分支。

## 8. 建议明确数组变长元素策略

当前协议标准规定数组信号仅支持固定长度数据类型。后续如要支持 `kString[]` 或 `kBinary[]`，建议明确：

1. 每个元素是否使用 `uint64 data_length + data`。
2. 数组内部是否允许元素级 `NULL`。
3. 元素级 `NULL` 使用独立 bitmap 还是特殊长度值。
4. DuckDB 导入时映射为 `LIST<VARCHAR>`、`LIST<BLOB>` 或其他结构。

## 9. 建议消除 EOF 标志歧义

当前协议使用文件末尾 8 字节固定值 `0xDEADBEEFDEADBEEF` 标识结束。解析器只能在准备读取下一帧时判断 EOF，不能在文件任意位置搜索 EOF。

风险：

- 如果文件在下一帧起始处只剩 8 字节且该值等于 EOF 标志，当前协议无法区分“正常 EOF”和“被截断但前 8 字节恰好等于 EOF 的损坏帧”。
- 缺少 `frame_size` 或 `frame_count` 时，EOF 缺失、截断和部分损坏文件的定位能力有限。

后续可选优化：

1. 增加 `frame_size`，使解析器能明确每帧边界。
2. 增加 `frame_count`，使读取可以按帧数结束，再校验 EOF。
3. 保留 EOF 只作为收尾校验，而不是唯一结束条件。
4. 如果继续以 EOF 为结束条件，需定义校验、转义或尾部结构来识别截断场景。

## 10. 建议明确时间信号约束

当前协议说明时间信号始终作为第一个信号记录，且数据帧中时间信号固定为 8 字节 `uint64`。建议标准层补充明确：

1. 第 0 个信号的 `signal_type` 必须为 `kUint64`。
2. 第 0 个信号的 `dimension_count` 必须为 0。
3. 每帧 bit 0 必须为 1。
4. 数据帧中 bitmap 后的固定 8 字节时间值就是第 0 个信号值；后续 `signal_data` 从信号 1 开始。

## 11. 建议明确错误恢复策略

当前协议适合默认拒绝损坏文件。后续如希望部分恢复，建议协议层定义：

- 帧校验和。
- 帧长度。
- 文件级帧索引。
- EOF 缺失但 `frame_count` 可信时是否允许 best-effort 导入。

## 12. 建议的新版本草案结构

如果升级为下一版协议，可考虑：

```text
FileHeader
SignalMeta[signal_count]
Frame[frame_count or until EOF]
EOF
```

示例文件头：

| 字段 | 大小 | 类型 | 说明 |
|---|---:|---|---|
| magic | 8 | bytes | `BINREC01` |
| version | 8 | uint64 | 协议版本，保持当前协议字段宽度 |
| flags | 4 | uint32 | 文件级 flags |
| header_size | 8 | uint64 | 数据区起始偏移 |
| signal_count | 8 | uint64 | 信号总数 |
| frame_count | 8 | uint64 | 帧数，可回填 |
| reserved | 32 | bytes | 预留 |

示例帧头：

| 字段 | 大小 | 类型 | 说明 |
|---|---:|---|---|
| frame_size | 8 | uint64 | 本帧总大小 |
| bitmap | `(signal_count + 7) / 8` | bytes | 信号存在位图 |
| time | 8 | uint64 | 时间信号 |
| signal_data | 可变 | bytes | 有值的非时间信号数据，从信号 1 开始 |

以上结构仅为建议，不属于当前协议标准。
