# 二进制数据记录方案设计

## 1. 概述

本文档定义仿真数据的二进制记录格式，以及在 Spring Boot 3 项目中的 DuckDB 落地方案。实现目标不是让 DuckDB 原生识别 `.bin` 文件，而是由 Java 服务解析二进制文件，通过 `org.duckdb:duckdb_jdbc:1.5.2.1` 写入 DuckDB 表，再使用 SQL 查询。

允许的升级路径：如果纯 Java 解析性能不足，可以引入 C++ 编译的动态库（`.so`）承担二进制解析；DuckDB 写入和查询仍优先通过 JDBC 完成。只有在确实需要 DuckDB 引擎内直接扫描 `.bin` 文件时，才实现 DuckDB C++ 扩展。

## 2. 设计目标

- 高效写入：按小端序直接记录原始二进制值，避免 CSV 文本解析成本。
- 稀疏记录：每帧用位图标记信号是否有值，无值信号在 DuckDB 中映射为 `NULL`。
- 支持数组信号：数组作为一个信号整体记录，不拆分成多个文件。
- 格式自描述：文件头保存信号类型、维度、名称和公式，解析器可据此建表。
- Java 可落地：首版实现以 Spring Boot 3 + `duckdb_jdbc:1.5.2.1` 为主。
- 可演进：格式带 magic、版本、header 长度、帧数和 flags，便于后续兼容。

## 3. 总体架构

```text
仿真程序/采集程序
        |
        v
二进制记录文件 .bin
        |
        v
Spring Boot Import Job
        |
        +-- BinaryRecordHeaderReader
        +-- BinaryRecordFrameReader
        +-- DuckDbSchemaMapper
        |
        v
DuckDBConnection.createAppender(...)
        |
        v
DuckDB 表 / SQL 查询
```

首版推荐流程：

1. Java 解析文件头。
2. 根据文件头生成 DuckDB 表结构。
3. Java 顺序读取数据帧。
4. 使用 `DuckDBAppender` 批量写入。
5. 用户通过普通 SQL 查询 DuckDB 表。

可选增强：

- 注册 Java table function，实现 `SELECT * FROM read_bin_record('/path/data.bin')`。
- C++ `.so` 提供高性能 parser，Java 通过 JNI/JNA/FFM 调用。
- DuckDB C++ 扩展实现引擎内 `read_bin_record()`，必要时再支持 replacement scan。

## 4. 文件结构 v1

文件分为三段：

```text
FileHeader
SignalMeta[signal_count]
Frame[frame_count]
```

所有多字节整数和浮点数均使用 Little Endian。

### 4.1 文件头结构

| 字段 | 大小 | 类型 | 说明 |
|---|---:|---|---|
| magic | 8 | bytes | 固定为 `BINREC01`，用于识别格式 |
| version | 4 | uint32 | 当前版本为 `1` |
| flags | 4 | uint32 | 文件级 flags，首版写 0 |
| header_size | 8 | uint64 | 从文件开头到数据区开始的总字节数 |
| signal_count | 8 | uint64 | 信号总数，包含时间信号 |
| frame_count | 8 | uint64 | 已写入帧数；流式写入中可先写 0，关闭文件时回填 |
| reserved | 32 | bytes | 预留，写 0 |

约束：

1. `magic` 不匹配时必须拒绝解析。
2. 未知 `version` 默认拒绝解析，除非 parser 显式声明兼容。
3. `header_size` 必须大于文件头固定大小，且小于等于文件大小。
4. `signal_count >= 1`。
5. 时间信号必须是第 0 个信号。

### 4.2 信号元信息结构

每个信号按定义顺序写入：

| 字段 | 大小 | 类型 | 说明 |
|---|---:|---|---|
| signal_type | 1 | uint8 | 数据类型枚举 |
| dimension_count | 4 | uint32 | 普通信号为 0，数组信号为维度数量 |
| dimensions | 4*N | uint32[] | 每个维度长度，仅当 `dimension_count > 0` |
| name_length | 8 | uint64 | 信号名称 UTF-8 字节长度 |
| name | 可变 | bytes | 信号名称 UTF-8 |
| formula_length | 4 | uint32 | 公式字符串 UTF-8 字节长度 |
| formula | 可变 | bytes | 公式字符串 UTF-8，可为空 |

约束：

1. `dimension_count = 0` 表示标量。
2. `dimension_count > 0` 时，所有 `dimensions[i] > 0`。
3. 数组元素总数为所有维度长度乘积。
4. `name` 可以包含任意 UTF-8 字符；建 DuckDB 表时必须做 SQL 标识符转义或规范化。
5. 原始信号名、公式和维度必须写入元数据表，避免列名规范化后丢失信息。

### 4.3 数据帧结构

每个时间帧按时间顺序追加：

| 字段 | 大小 | 类型 | 说明 |
|---|---:|---|---|
| frame_size | 8 | uint64 | 本帧总大小，包含 `frame_size` 字段自身 |
| bitmap | `(signal_count + 7) / 8` | bytes | 信号存在位图 |
| signal_data | 可变 | bytes | 有值信号的数据，按信号定义顺序排列 |

位图规则：

```text
signal i 是否有值 = (bitmap[i / 8] & (1 << (i % 8))) != 0
```

约束：

1. 位图使用 LSB first，即 bit 0 对应信号 0。
2. 时间信号为第 0 个信号，首版要求每帧 `bit 0 = 1`。
3. bitmap 为 0 的信号不写入任何数据，导入 DuckDB 时写 `NULL`。
4. bitmap 为 1 的信号按照信号定义顺序连续写入。
5. `frame_size` 使 parser 能跳过未知帧、校验帧边界，并为后续建立帧索引提供基础。
6. EOF 必须落在完整帧边界；如果最后一帧不完整，默认拒绝导入。

## 5. 数据类型

### 5.1 基本类型

| 枚举值 | 类型值 | 大小 | DuckDB 推荐类型 | 说明 |
|---|---:|---:|---|---|
| kUint64 | 0 | 8 | `UBIGINT` | 64 位无符号整数 |
| kUint32 | 1 | 4 | `UINTEGER` | 32 位无符号整数 |
| kUint16 | 2 | 2 | `USMALLINT` | 16 位无符号整数 |
| kUint8 | 3 | 1 | `UTINYINT` | 8 位无符号整数 |
| kInt64 | 4 | 8 | `BIGINT` | 64 位有符号整数 |
| kInt32 | 5 | 4 | `INTEGER` | 32 位有符号整数 |
| kInt16 | 6 | 2 | `SMALLINT` | 16 位有符号整数 |
| kInt8 | 7 | 1 | `TINYINT` | 8 位有符号整数 |
| kBool | 8 | 1 | `BOOLEAN` | 0=false，非 0=true |
| kEnum | 9 | 8 | `BIGINT` | 首版按整数码存储 |
| kFloat | 10 | 4 | `FLOAT` | IEEE 754 float |
| kDouble | 11 | 8 | `DOUBLE` | IEEE 754 double |
| kString | 12 | 可变 | `VARCHAR` | UTF-8 |
| kBinary | 13 | 可变 | `BLOB` | 原始 bytes |
| kComplex32 | 14 | 8 | `STRUCT(re FLOAT, im FLOAT)` | 两个 float：实部、虚部 |
| kComplex64 | 15 | 16 | `STRUCT(re DOUBLE, im DOUBLE)` | 两个 double：实部、虚部 |

### 5.2 固定长度类型

除 `kString`、`kBinary` 外，其他类型为固定长度。固定长度标量按上表大小直接写入。

复数类型按以下顺序写入：

```text
kComplex32: float re, float im
kComplex64: double re, double im
```

### 5.3 变长类型

`kString` 和 `kBinary` 使用长度前缀：

| 字段 | 大小 | 类型 | 说明 |
|---|---:|---|---|
| data_length | 8 | uint64 | 数据字节长度 |
| data | 可变 | bytes | 实际数据 |

约束：

1. `kString` 的 `data` 必须是合法 UTF-8。
2. `kBinary` 不做编码解释。
3. `data_length = 0` 表示空字符串或空二进制，不表示 NULL。
4. NULL 只能由帧 bitmap 表示。

### 5.4 数组类型

数组信号由信号元信息中的 `dimension_count` 和 `dimensions` 定义。数组数据按行主序（row-major）连续写入，每个元素按其基础类型格式写入。

约束：

1. 数组元素个数固定为 `product(dimensions)`。
2. 数组整体是否存在由帧 bitmap 控制。
3. 首版不支持数组内部元素级 NULL。
4. 数组元素为 `kString` 或 `kBinary` 时，每个元素都按 `uint64 data_length + data` 格式写入。
5. 多维数组导入 DuckDB 时首选嵌套 `LIST`；维度信息另存元数据表。

## 6. DuckDB 表设计

### 6.1 数据表

每个二进制文件导入到一张 DuckDB 数据表，表名由业务侧指定，不能直接使用未转义文件名。

示例：

```sql
CREATE TABLE sim_record (
    time_ns UBIGINT NOT NULL,
    speed DOUBLE,
    voltage FLOAT,
    flags UTINYINT,
    spectrum DOUBLE[],
    payload BLOB
);
```

列名规则：

1. 第 0 个时间信号建议映射为 `time_ns` 或原始信号名的规范化版本。
2. 原始信号名必须保存到元数据表。
3. DuckDB SQL 中的列名必须转义或规范化，避免关键字、空格、重复名、大小写冲突。
4. 重复列名使用稳定后缀，例如 `_1`、`_2`。

### 6.2 元数据表

建议每次导入同时写入元数据表：

```sql
CREATE TABLE bin_record_signal_meta (
    import_id VARCHAR,
    signal_index UBIGINT,
    original_name VARCHAR,
    column_name VARCHAR,
    signal_type UTINYINT,
    dimension_count UINTEGER,
    dimensions_json VARCHAR,
    formula VARCHAR,
    duckdb_type VARCHAR
);
```

如果需要文件级信息：

```sql
CREATE TABLE bin_record_file_meta (
    import_id VARCHAR,
    file_path VARCHAR,
    format_version UINTEGER,
    signal_count UBIGINT,
    frame_count UBIGINT,
    imported_at TIMESTAMP
);
```

## 7. Java/Spring Boot 实现设计

### 7.1 模块划分

| 模块 | 职责 |
|---|---|
| `BinaryRecordHeaderReader` | 校验 magic/version，读取 header 和信号元信息 |
| `BinaryRecordFrameReader` | 顺序读取帧，解析 bitmap 和信号值 |
| `SignalTypeMapper` | 二进制类型到 DuckDB SQL 类型、Java 值类型的映射 |
| `DuckDbImportService` | 建表、写元数据、使用 Appender 导入 |
| `ColumnNameNormalizer` | 生成安全且稳定的 DuckDB 列名 |
| `ImportValidationService` | 校验文件大小、帧边界、信号数量、维度乘积等 |

### 7.2 Appender 写入流程

伪代码：

```java
try (Connection raw = dataSource.getConnection()) {
    DuckDBConnection conn = raw.unwrap(DuckDBConnection.class);
    createDataTable(conn, schema);
    writeMetadata(conn, importId, schema);

    try (DuckDBAppender appender = conn.createAppender(tableName)) {
        while (reader.hasNextFrame()) {
            Frame frame = reader.nextFrame();
            appender.beginRow();
            for (int i = 0; i < schema.signalCount(); i++) {
                if (!frame.hasValue(i)) {
                    appender.appendNull();
                    continue;
                }
                appendSignalValue(appender, schema.signal(i), frame.value(i));
            }
            appender.endRow();
        }
        appender.flush();
    }
}
```

实现注意：

1. `DuckDBAppender` 不是通用多线程 writer，单个 appender 应限制在一个导入线程使用。
2. 大文件导入应分批 flush，并记录导入进度。
3. 导入失败时应删除半成品表，或写入 `import_status` 标记。
4. 无符号整数超出 Java 有符号范围时要保留原始位模式或使用 `BigInteger` 策略。
5. `kComplex32/kComplex64` 首版如果 struct 写入复杂，可配置降级为数组或两列。

### 7.3 Java table function 选项

`duckdb_jdbc:1.5.2.1` 提供 `DuckDBFunctions.tableFunction()` 和 `DuckDBTableFunctionBuilder`。可选实现：

```sql
SELECT * FROM read_bin_record('/data/sim.bin');
```

适用场景：

- 临时分析，不希望先导入表。
- 文件较小或可接受顺序扫描。
- 需要用 SQL 统一访问多个外部文件。

限制：

- 这不是 replacement scan，不能让 `SELECT * FROM 'data.bin'` 自动成立。
- Java 回调性能需压测。
- 状态管理和异常处理比 Appender 导入更复杂。

## 8. C++ 动态库混合方案

如果纯 Java parser 成为瓶颈，可引入 `libbinrecord.so`：

```text
Java ImportService
        |
        v
JNI/JNA/FFM
        |
        v
libbinrecord.so
        |
        v
批量解码结果
        |
        v
DuckDBAppender
```

建议边界：

1. C++ `.so` 只负责二进制格式校验和批量解码。
2. DuckDB 连接、事务、建表、Appender 写入仍由 Java 管理。
3. C++ 返回批量列式或行式 buffer，Java 负责映射到 Appender。
4. 先定义稳定 C ABI，避免直接暴露 C++ 类。

只有当需求明确为 DuckDB 引擎内直接扫描 `.bin` 文件时，再做 DuckDB C++ 扩展。

## 9. 文件扩展名

二进制记录文件使用 `.bin` 作为业务约定扩展名。

注意：

1. `.bin` 只是文件命名约定，不表示 DuckDB 会自动识别。
2. DuckDB 当前不会因为扩展名是 `.bin` 就执行 replacement scan。
3. 首版查询方式应是先导入表再查表，或显式调用 `read_bin_record(path)`。

## 10. 校验与错误处理

导入前必须校验：

- magic 是否正确。
- version 是否支持。
- `header_size` 是否合理。
- `signal_count` 是否与元信息数量一致。
- 时间信号是否存在且为第 0 个信号。
- 数组维度乘积是否溢出。
- 每帧 `frame_size` 是否至少覆盖 bitmap。
- 每帧实际读取字节数是否等于 `frame_size`。
- EOF 是否落在完整帧边界。

错误处理策略：

| 错误 | 策略 |
|---|---|
| magic/version 不匹配 | 拒绝导入 |
| header 损坏 | 拒绝导入 |
| 中间帧损坏 | 默认拒绝导入；可选容错模式跳过损坏帧 |
| 最后一帧不完整 | 默认拒绝导入 |
| UTF-8 非法 | 拒绝导入或按配置替换 |
| DuckDB 写入失败 | 回滚事务或删除半成品表 |

## 11. 测试计划

必须覆盖：

1. 只含时间信号的最小文件。
2. 所有基本类型各一列。
3. 稀疏帧：部分信号 bitmap 为 0，导入后为 `NULL`。
4. `kString`、`kBinary` 空值和非空值。
5. 一维数组、二维数组。
6. 数组元素为变长类型。
7. 复数类型。
8. 重复/非法 SQL 列名规范化。
9. 文件截断、frame_size 错误、header_size 错误。
10. 大文件批量导入性能测试。

## 12. 与 CSV 对比

| 特性 | CSV | 二进制格式 |
|---|---|---|
| 存储空间 | 较大 | 较小 |
| 写入性能 | 需要文本格式化 | 直接写二进制 |
| 读取性能 | 需要文本解析 | 顺序二进制解析 |
| 稀疏数据 | 空字段或额外约定 | bitmap 原生表达 |
| 数组信号 | 通常需要拆列或编码为字符串 | 原生连续存储 |
| 人类可读性 | 高 | 低 |
| DuckDB 直接读取 | 内置支持 CSV | 需 Java 导入、Java table function 或 C++ 扩展 |
