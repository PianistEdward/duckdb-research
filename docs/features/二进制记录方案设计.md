# 二进制数据记录方案设计

## 1. 概述

本文档描述了二进制数据记录的详细方案，该方案将仿真数据以二进制格式存储到文件，相比CSV格式具有更高的读写效率和更小的存储空间利用率。

## 2. 设计目标

- 
- 高效读写性能：直接存储原始二进制数据，无需解析转换
- 支持数组类型信号统一记录，不拆分
- 文件格式清晰，便于后续解析
- 支持多种数据类型的统一处理

## 3. 文件结构

二进制文件整体分为两部分：
1. 文件头（Header）和数据区（Data）

### 3.1 文件头结构

文件头位于文件开头，包含版本号和信号元信息。

| 字段 | 大小（字节） | 说明 |
|-----|-------------|-----|
| 版本信息 | 8 | 文件格式版本号，用于向后兼容 |
| 信号总数 | 8 | 记录的信号总个数 |
| 信号1信息 | 可变 | 包含信号类型、维度数量、维度长度数组、名称等 |
| ... | ... | ... |
| 信号N信息 | 可变 | 最后一个信号的元信息 |

### 3.2 信号元信息结构

每个信号的元信息按以下顺序存储：

| 字段 | 大小（字节） | 说明 |
|-----|-------------|-----|
| 信号类型 | 1 | 信号的数据类型枚举值 |
| 维度数量 | 4 | uint32，表示数组的维度数量，普通信号为0 |
| 维度长度数组 | 4*N | uint32数组，每个维度的长度，仅在维度数量>0时存在 |
| 名称长度 | 8 | uint64，信号名称的字节长度 |
| 信号名称 | 可变 | 信号名称的UTF-8编码 |
| 公式长度 | 4 | uint32，公式字符串的字节长度 |
| 公式字符串 | 可变 | 公式字符串的UTF-8编码，仅在公式长度>0时存在 |

注意：
1. 时间信号始终作为第一个信号记录。
2. 对于普通信号（非数组），维度数量为0，没有维度长度数组。
3. 元素总数为各维度长度的乘积。

### 3.3 数据区结构

数据区紧随文件头之后，按时间顺序存储实际数据。每个时间帧格式如下：

| 字段 | 大小（字节） | 说明 |
|-----|-------------|-----|
| 信号位图 | (N+7)/8 | 每一位对应一个信号，1表示本帧有数据，0表示无数据。位图包含时间信号（第0位始终为1） |
| 时间信号 | 8 | 时间信号，固定8字节uint64 |
| 信号2数据（若有） | 可变 | 第二个信号的二进制数据 |
| ... | ... | ... |
| 信号M数据（若有） | 可变 | 第M个有数据信号的二进制数据 |

注意：
1. 时间信号始终有数据，对应位图第0位为1
2. 有数据的信号严格按信号定义顺序排列
3. N为信号总数，包括时间信号

## 4. 数据类型处理

### 4.1 基本数据类型

基本数据类型直接按其固有大小存储：

| 枚举值 | 类型值 | 大小（字节） | 说明 |
|-------|--------|-------------|-----|
| kUint64 | 0 | 8 | 64位无符号整数 |
| kUint32 | 1 | 4 | 32位无符号整数 |
| kUint16 | 2 | 2 | 16位无符号整数 |
| kUint8 | 3 | 1 | 8位无符号整数 |
| kInt64 | 4 | 8 | 64位有符号整数 |
| kInt32 | 5 | 4 | 32位有符号整数 |
| kInt16 | 6 | 2 | 16位有符号整数 |
| kInt8 | 7 | 1 | 8位有符号整数 |
| kBool | 8 | 1 | 布尔类型 |
| kEnum | 9 | 8 | 枚举类型 |
| kFloat | 10 | 4 | 32位浮点数 |
| kDouble | 11 | 8 | 64位浮点数 |
| kString | 12 | 可变 | 字符串类型 |
| kBinary | 13 | 可变 | 二进制数据 |
| kComplex32 | 14 | 8 | 32位复数（float+float） |
| kComplex64 | 15 | 16 | 64位复数（double+double） |

### 4.1 固定长度数据类型

除了 `kString` 和 `kBinary` 外，其他类型均为固定长度，按上表大小直接存储原始二进制数据。

### 4.2 变长数据类型

对于 `kString` 和 `kBinary` 类型，数据格式为：

| 字段 | 大小（字节） | 说明 |
|-----|-------------|-----|
| 数据长度 | 8 | uint64，表示实际数据的字节长度 |
| 实际数据 | 可变 | 实际的二进制数据，长度由前一字段指定 |

### 4.3 数组类型

数组类型按顺序连续存储每个元素，元素个数在文件头中已指定。每个元素按照其对应的数据类型存储。

## 5. 实现要点

1. 文件头在文件初始化时写入一次，后续只追加数据区
2. 时间信号始终作为第一个信号记录
3. 数据按时间戳顺序连续写入
4. 使用小端序（Little Endian）存储多字节数据
5. 字符串使用UTF-8编码

## 6. 与CSV对比

| 特性 | CSV格式 | 二进制格式 |
|-----|---------|-----------|
| 存储空间 | 较大（文本格式） | 较小（原始二进制） |
| 读写性能 | 较慢（需要解析） | 较快（直接读写） |
| 可读性 | 高（可读文本） | 低（需解析） |
| 数组处理 | 需拆分文件 | 统一记录 |

## 7. 文件扩展名

二进制记录文件使用 `.bin` 作为扩展名，便于与 `.csv` 文件区分。
