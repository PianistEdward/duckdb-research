# 二进制数据记录方案设计

## 1. 概述

本文档基于 `docs/features/二进制记录协议标准.md` 描述二进制记录文件在 Spring Boot 3 + DuckDB 中的落地设计。当前协议标准是本文档的实现依据；如本文档与协议标准冲突，以协议标准为准。

当前目标不是让 DuckDB 原生识别 `.bin` 文件，而是由 Java 服务解析二进制记录文件，再通过 `org.duckdb:duckdb_jdbc:1.5.2.1` 写入 DuckDB 表，后续使用 SQL 查询。

允许的升级路径：

1. 首版使用 Java 顺序解析协议标准文件。
2. 如果 Java 解析性能不足，可引入 C++ 动态库（`.so`）只负责二进制解析和批量解码。
3. 只有在确实需要 DuckDB 引擎内直接扫描 `.bin` 文件时，才实现 DuckDB C++ 扩展。

## 2. 设计目标

- 严格兼容当前二进制记录协议标准。
- 使用小端序直接读写原始二进制值，避免 CSV 文本解析成本。
- 使用帧级位图表达稀疏数据，无值信号导入 DuckDB 时写入 `NULL`。
- 数组信号作为一个信号整体记录，不拆分为多个文件。
- 从文件头读取信号类型、维度、名称和公式，自动生成 DuckDB 表结构。
- 保留原始信号名、公式、维度等元数据，避免 SQL 列名规范化造成信息丢失。

## 3. 总体架构

```text
仿真程序 / 采集程序
        |
        v
二进制记录文件 .bin
        |
        v
Spring Boot Import Job
        |
        +-- BinaryRecordHeaderReader
        +-- BinaryRecordFrameReader
        +-- DuckDbSchemaMapper
        +-- DuckDbImportService
        |
        v
DuckDBConnection.createAppender(...)
        |
        v
DuckDB 数据表 / 元数据表 / SQL 查询
```

首版推荐流程：

1. Java 读取并校验文件头。
2. 根据信号元信息生成 DuckDB 数据表和元数据表。
3. Java 按时间顺序读取数据帧。
4. 使用 `DuckDBAppender` 批量写入 DuckDB。
5. 用户查询 DuckDB 表，而不是直接查询 `.bin` 文件。

可选增强：

- 注册 Java table function，支持 `SELECT * FROM read_bin_record('/path/data.bin')`。
- 引入 C++ `.so` 作为高性能 parser，Java 通过 JNI/JNA/FFM 调用。
- 实现 DuckDB C++ 扩展，在引擎内提供 `read_bin_record()`。

## 4. 当前协议文件结构

协议标准定义的二进制文件由三部分组成：

```text
Header
DataFrame*
EOF
```

所有多字节整数和浮点数均使用 Little Endian。字符串使用 UTF-8 编码。

### 4.1 文件头结构

文件头位于文件开头，包含版本信息、信号总数和每个信号的元信息。

| 字段 | 大小 | 类型 | 说明 |
|---|---:|---|---|
| version | 8 | uint64 | 文件格式版本号 |
| signal_count | 8 | uint64 | 信号总数，包含时间信号 |
| signal_meta | 可变 | SignalMeta[] | 按信号定义顺序写入 |

约束：

1. `signal_count >= 1`。
2. 时间信号始终作为第 0 个信号。
3. 协议标准当前未定义 magic、header_size 或 frame_count；解析器不能依赖这些字段。

### 4.2 信号元信息结构

每个信号按以下顺序写入：

| 字段 | 大小 | 类型 | 说明 |
|---|---:|---|---|
| signal_type | 1 | uint8 | 数据类型枚举 |
| dimension_count | 4 | uint32 | 标量为 0，数组信号为维度数量 |
| dimensions | 4*N | uint32[] | 每个维度长度，仅当 `dimension_count > 0` |
| name_length | 8 | uint64 | 信号名称 UTF-8 字节长度 |
| name | 可变 | bytes | 信号名称 UTF-8 |
| formula_length | 4 | uint32 | 公式字符串 UTF-8 字节长度 |
| formula | 可变 | bytes | 公式字符串 UTF-8，可为空 |

约束：

1. `dimension_count = 0` 表示标量。
2. `dimension_count > 0` 时，所有维度长度必须大于 0。
3. 数组元素总数为所有维度长度的乘积。
4. 建 DuckDB 表时必须转义或规范化信号名称。
5. 原始信号名、公式和维度必须写入元数据表。

### 4.3 数据帧结构

数据区紧随文件头之后，按时间顺序存储数据帧。每帧结构如下：

| 字段 | 大小 | 类型 | 说明 |
|---|---:|---|---|
| bitmap | `(signal_count + 7) / 8` | bytes | 信号存在位图 |
| time | 8 | uint64 | 时间信号，固定 8 字节 |
| signal_data | 可变 | bytes | 有值信号的数据，按信号定义顺序写入 |

位图规则：

```text
signal i 是否有值 = (bitmap[i / 8] & (1 << (i % 8))) != 0
```

约束：

1. 位图使用 LSB first，即 bit 0 对应信号 0。
2. 时间信号为第 0 个信号，首版要求每帧 bit 0 为 1。
3. bitmap 为 0 的信号不写入任何数据，导入 DuckDB 时写 `NULL`。
4. bitmap 为 1 的信号按照信号定义顺序连续写入。
5. 协议标准当前未定义 `frame_size`；解析器必须通过信号类型和维度顺序消费每帧数据。

### 4.4 文件结束标志

所有时间帧之后写入 EOF 标志：

| 字段 | 大小 | 类型 | 说明 |
|---|---:|---|---|
| eof_marker | 8 | uint64 | 固定值 `0xDEADBEEFDEADBEEF` |

解析策略：

1. 当读取到完整 EOF 标志时，导入正常结束。
2. EOF 必须出现在完整帧之后。
3. 如果解析帧过程中遇到文件结束、字段长度越界或无法消费到 EOF，默认拒绝导入。
4. 因协议标准没有帧长度，EOF 值理论上可能出现在数据内容中；解析器只能在“准备读取下一帧开头”时判断 EOF，而不能在任意位置扫描 EOF。

## 5. 数据类型映射

### 5.1 基本类型

| 枚举值 | 类型值 | 大小 | DuckDB 推荐类型 | 说明 |
|---|---:|---:|---|---|
| `kUint64` | 0 | 8 | `UBIGINT` | 64 位无符号整数 |
| `kUint32` | 1 | 4 | `UINTEGER` | 32 位无符号整数 |
| `kUint16` | 2 | 2 | `USMALLINT` | 16 位无符号整数 |
| `kUint8` | 3 | 1 | `UTINYINT` | 8 位无符号整数 |
| `kInt64` | 4 | 8 | `BIGINT` | 64 位有符号整数 |
| `kInt32` | 5 | 4 | `INTEGER` | 32 位有符号整数 |
| `kInt16` | 6 | 2 | `SMALLINT` | 16 位有符号整数 |
| `kInt8` | 7 | 1 | `TINYINT` | 8 位有符号整数 |
| `kBool` | 8 | 1 | `BOOLEAN` | 0=false，非 0=true |
| `kEnum` | 9 | 8 | `BIGINT` | 首版按整数码存储 |
| `kFloat` | 10 | 4 | `FLOAT` | IEEE 754 float |
| `kDouble` | 11 | 8 | `DOUBLE` | IEEE 754 double |
| `kString` | 12 | 可变 | `VARCHAR` | UTF-8 |
| `kBinary` | 13 | 可变 | `BLOB` | 原始 bytes |
| `kComplex32` | 14 | 8 | `STRUCT(re FLOAT, im FLOAT)` | 两个 float：实部、虚部 |
| `kComplex64` | 15 | 16 | `STRUCT(re DOUBLE, im DOUBLE)` | 两个 double：实部、虚部 |

### 5.2 固定长度类型

除 `kString` 和 `kBinary` 外，其他类型为固定长度，按上表大小直接写入原始二进制数据。

复数类型按以下顺序写入：

```text
kComplex32: float re, float im
kComplex64: double re, double im
```

### 5.3 变长类型

`kString` 和 `kBinary` 使用长度前缀：

| 字段 | 大小 | 类型 | 说明 |
|---|---:|---|---|
| data_length | 8 | uint64 | 数据字节长度 |
| data | 可变 | bytes | 实际数据 |

约束：

1. `kString` 的 `data` 必须是合法 UTF-8。
2. `kBinary` 不做编码解释。
3. `data_length = 0` 表示空字符串或空二进制，不表示 `NULL`。
4. `NULL` 只能由帧 bitmap 表示。

### 5.4 数组类型

数组信号由信号元信息中的 `dimension_count` 和 `dimensions` 定义。数组数据按信号基础类型连续写入。

当前协议标准约束：

1. 数组信号仅支持固定长度数据类型。
2. 数组元素个数固定为 `product(dimensions)`。
3. 数组整体是否存在由帧 bitmap 控制。
4. 首版不支持数组内部元素级 `NULL`。
5. 数组导入 DuckDB 时首选 `LIST` 或嵌套 `LIST`，维度信息另存元数据表。

## 6. DuckDB 表设计

### 6.1 数据表

每个二进制文件导入到一张 DuckDB 数据表，表名由业务侧指定，不能直接使用未转义文件名。

示例：

```sql
CREATE TABLE sim_record (
    time_ns UBIGINT NOT NULL,
    speed DOUBLE,
    voltage FLOAT,
    flags UTINYINT,
    spectrum DOUBLE[],
    payload BLOB
);
```

列名规则：

1. 第 0 个时间信号建议映射为 `time_ns` 或原始信号名的规范化版本。
2. 原始信号名必须保存到元数据表。
3. DuckDB SQL 中的列名必须转义或规范化，避免关键字、空格、重复名、大小写冲突。
4. 重复列名使用稳定后缀，例如 `_1`、`_2`。

### 6.2 元数据表

每次导入同时写入信号元数据表：

```sql
CREATE TABLE bin_record_signal_meta (
    import_id VARCHAR,
    signal_index UBIGINT,
    original_name VARCHAR,
    column_name VARCHAR,
    signal_type UTINYINT,
    dimension_count UINTEGER,
    dimensions_json VARCHAR,
    formula VARCHAR,
    duckdb_type VARCHAR
);
```

文件级信息建议写入：

```sql
CREATE TABLE bin_record_file_meta (
    import_id VARCHAR,
    file_path VARCHAR,
    format_version UBIGINT,
    signal_count UBIGINT,
    imported_frame_count UBIGINT,
    eof_valid BOOLEAN,
    imported_at TIMESTAMP
);
```

## 7. Java/Spring Boot 实现设计

### 7.1 模块划分

| 模块 | 职责 |
|---|---|
| `BinaryRecordHeaderReader` | 读取 version、signal_count 和信号元信息 |
| `BinaryRecordFrameReader` | 顺序读取帧，解析 bitmap 和信号值，识别 EOF |
| `SignalTypeMapper` | 二进制类型到 DuckDB SQL 类型、Java 值类型的映射 |
| `DuckDbSchemaMapper` | 生成建表 SQL、列名规范化和元数据记录 |
| `DuckDbImportService` | 建表、写元数据、使用 Appender 导入 |
| `ColumnNameNormalizer` | 生成安全且稳定的 DuckDB 列名 |
| `ImportValidationService` | 校验文件大小、字段长度、信号数量、维度乘积等 |

### 7.2 Appender 写入流程

伪代码：

```java
try (Connection raw = dataSource.getConnection()) {
    DuckDBConnection conn = raw.unwrap(DuckDBConnection.class);
    createDataTable(conn, schema);
    writeMetadata(conn, importId, schema);

    try (DuckDBAppender appender = conn.createAppender(tableName)) {
        while (reader.hasNextFrame()) {
            Frame frame = reader.nextFrame();
            appender.beginRow();
            for (int i = 0; i < schema.signalCount(); i++) {
                if (!frame.hasValue(i)) {
                    appender.appendNull();
                    continue;
                }
                appendSignalValue(appender, schema.signal(i), frame.value(i));
            }
            appender.endRow();
        }
        appender.flush();
    }
}
```

实现注意：

1. `DuckDBAppender` 不是通用多线程 writer，单个 appender 限制在一个导入线程使用。
2. 大文件导入应分批 flush，并记录导入进度。
3. 导入失败时应回滚事务、删除半成品表，或写入 `import_status` 标记。
4. 无符号整数超出 Java 有符号范围时要保留原始位模式或使用 `BigInteger` 策略。
5. `kComplex32`/`kComplex64` 如果 struct 写入复杂，可配置降级为长度为 2 的数组或两列。

### 7.3 EOF 驱动的读取流程

协议标准没有 `frame_count` 和 `frame_size`，读取器应按以下策略工作：

1. 读取文件头后，当前位置即为数据区开始。
2. 读取下一帧前先尝试读取 8 字节并按小端序判断是否等于 EOF 标志。
3. 如果等于 EOF 且 8 字节后到达文件末尾，结束导入。
4. 如果不是 EOF，或等于 EOF 但后续仍有字节，这 8 字节属于帧开头的 bitmap 前缀，需要放回或使用缓冲区继续解析本帧。
5. 按 bitmap 和信号定义顺序消费本帧所有字段。
6. 如果字段读取越界、变长长度超过剩余文件大小或 EOF 不在帧边界，拒绝导入。

当 bitmap 字节数小于 8 时，实现可以使用小型前读缓冲区；不要在文件任意位置搜索 EOF 标志。若业务允许 EOF 后追加校验信息或尾部元数据，必须先升级协议标准，否则首版解析器按 EOF 为文件最后 8 字节处理。

### 7.4 Java table function 选项

`duckdb_jdbc:1.5.2.1` 提供 `DuckDBFunctions.tableFunction()` 和 `DuckDBTableFunctionBuilder`。可选实现：

```sql
SELECT * FROM read_bin_record('/data/sim.bin');
```

适用场景：

- 临时分析，不希望先导入表。
- 文件较小或可接受顺序扫描。
- 需要用 SQL 统一访问多个外部文件。

限制：

- 这不是 replacement scan，不能让 `SELECT * FROM 'data.bin'` 自动成立。
- Java 回调性能需压测。
- 状态管理和异常处理比 Appender 导入更复杂。

## 8. C++ 动态库混合方案

如果纯 Java parser 成为瓶颈，可引入 `libbinrecord.so`：

```text
Java ImportService
        |
        v
JNI/JNA/FFM
        |
        v
libbinrecord.so
        |
        v
批量解码结果
        |
        v
DuckDBAppender
```

建议边界：

1. C++ `.so` 只负责当前协议标准的格式校验和批量解码。
2. DuckDB 连接、事务、建表、Appender 写入仍由 Java 管理。
3. C++ 返回批量列式或行式 buffer，Java 负责映射到 Appender。
4. 先定义稳定 C ABI，避免直接暴露 C++ 类。

只有当需求明确为 DuckDB 引擎内直接扫描 `.bin` 文件时，再做 DuckDB C++ 扩展。

## 9. 文件扩展名

二进制记录文件使用 `.bin` 作为业务约定扩展名。

注意：

1. `.bin` 只是文件命名约定，不表示 DuckDB 会自动识别。
2. DuckDB 当前不会因为扩展名是 `.bin` 就执行 replacement scan。
3. 首版查询方式应是先导入表再查表，或显式调用 `read_bin_record(path)`。

## 10. 校验与错误处理

导入前和导入过程中必须校验：

- `version` 是否支持。
- `signal_count` 是否大于 0。
- 时间信号是否存在且为第 0 个信号。
- 信号类型枚举是否在支持范围内。
- `name_length`、`formula_length`、`data_length` 是否未越界。
- 数组维度乘积是否溢出。
- 数组信号是否只使用固定长度数据类型。
- 每帧 bit 0 是否为 1。
- EOF 是否出现在完整帧边界。

错误处理策略：

| 错误 | 策略 |
|---|---|
| version 不支持 | 拒绝导入 |
| header 损坏 | 拒绝导入 |
| 中间帧损坏 | 默认拒绝导入；不建议静默跳过 |
| 最后一帧不完整 | 默认拒绝导入 |
| UTF-8 非法 | 拒绝导入或按配置替换 |
| EOF 缺失或位置错误 | 拒绝导入 |
| DuckDB 写入失败 | 回滚事务或删除半成品表 |

## 11. 测试计划

必须覆盖：

1. 只含时间信号的最小文件。
2. 所有基本类型各一列。
3. 稀疏帧：部分信号 bitmap 为 0，导入后为 `NULL`。
4. `kString`、`kBinary` 空值和非空值。
5. 一维数组、二维数组。
6. 数组元素为固定长度类型。
7. 数组信号错误使用 `kString` 或 `kBinary` 时拒绝导入。
8. 复数类型。
9. 重复/非法 SQL 列名规范化。
10. 文件截断、长度字段越界、EOF 缺失或 EOF 不在帧边界。
11. 大文件批量导入性能测试。

## 12. 与 CSV 对比

| 特性 | CSV | 二进制格式 |
|---|---|---|
| 存储空间 | 较大 | 较小 |
| 写入性能 | 需要文本格式化 | 直接写二进制 |
| 读取性能 | 需要文本解析 | 顺序二进制解析 |
| 稀疏数据 | 空字段或额外约定 | bitmap 原生表达 |
| 数组信号 | 通常需要拆列或编码为字符串 | 原生连续存储 |
| 人类可读性 | 高 | 低 |
| DuckDB 直接读取 | 内置支持 CSV | 需 Java 导入、Java table function 或 C++ 扩展 |
